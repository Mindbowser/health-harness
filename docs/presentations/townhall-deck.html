Step 1 · /align — together

PM + Engineer agree — and it becomes one story

🧑‍💼 The PM brings

the what & why — business intent

  • Goal: a patient can view their lab results in the portal
  • Rules: see only your own · newest first · never someone else’s

👩‍💻 The Engineer brings

the how + safety — grounded in the code

  • Feasibility: filter the existing results table by the signed-in patient
  • Flags HIPAA: results are PHI → audit every view · keep values out of logs

↓  /align turns the conversation into one shared, testable story

✓ Story SAMPLE-204 In Refinement

Patient can view their lab results in the portal

A signed-in patient sees a list of their own lab results, newest first. All data synthetic.

Acceptance criteria

  • AC-1A patient sees only their own results — never another patient’s🧑‍💼 PM
  • AC-2Results are shown newest first🧑‍💼 PM
  • AC-3Only provider-released results show; pending ones are hidden🧑‍💼 PM
  • AC-4Every view is audit-logged — who · what · when🛡️ HIPAA
  • AC-5Logs never contain the result values🛡️ HIPAA

Step 2 · /tdd — how we build

Test-Driven Development — and why it fits AI

Write the test first → make it pass → clean up. One behaviour at a time. Here is AC-1 of SAMPLE-204:

🔴 RED · write the test first
test('AC-1: only own results', () => {
  login(Raj)
  expect(results())
    .not.toContain(Jane.result)
})
// ✗ fails — Jane’s row leaks
🟢 GREEN · least code to pass
results = labResults
  .where({ patientId: user.id })

// ✓ the test passes
🔵 REFACTOR · clean up, stay green
function resultsForPatient(id){
  return labResults
    .where({ patientId: id })
}
// ✓ tests still green

Why TDD fits AI especially well

  • Proof, not vibes. AI writes code fast — the test proves it actually works, instead of just looking right.
  • No “fake green.” Writing the test first stops the AI declaring “done” without a real, passing check.
  • An executable spec. Each acceptance criterion becomes a test the AI can’t drift away from.
  • A self-correcting loop. The gate runs after every change, so the AI fixes its own mistakes — your tests set the quality ceiling.

The validation layer

What we check before anything ships

Automatic and exact — pass or fail, no guessing. If a check fails, it stops and fixes it.

📝 Acceptance-criteria coverage

Every criterion agreed in /align is pinned by a real test.

✔️ Green gate

Tests + types + lint all pass — re-run every change, never skipped.

🔒 PHI / PII redaction

Code, tickets & PRs scanned at the boundary; real patient data, identifiers or secrets are blocked.

🧾 Audit logging

PHI reads, writes & denied access recorded — who · what · when · outcome. The HIPAA trail.

🧹 Safe logging

Logs carry a record id, never PHI/PII values.

Plus lighter guards: timezone-safe date/time · conventional, ticket-keyed commits · feature-branch hygiene.

Where this goes next

Closing the loop — with QA raising the ceiling

🧪 QA deepens the tests

QA enhances the safety net — broader coverage and real end-to-end checks (e.g. Playwright) on top of the unit tests.

🔁 Close the loop from CI

CI failures feed straight back to the agent; fixes are re-verified automatically — less manual back-and-forth.

🎫 Automate the busywork

Ticket status, worklogs and hand-offs flow on their own — people spend time on judgment, not bookkeeping.

The flow becomes a system that improves itself — QA shifts from gate-keeping to raising the quality ceiling for everyone.

⚠️ Spend your attention where the machine is weakest

An LLM can hallucinate — in SAMPLE-204 it might invent a lab value that was never returned, or quietly mis-apply the access rule and show Raj someone else’s row. That’s exactly where you look: the AC-1 test catches the access slip, the audit + redaction checks catch PHI — and you own the judgment. Clear mental model + harness = your time goes to what only a human can do.
